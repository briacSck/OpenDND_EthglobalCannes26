# OpenD&D — Hackathon PRD / Technical Brief
Version: Cannes 2026 MVP


## 1. Product definition


OpenD&D is a mobile-first system that turns real-world outings into AI-generated narrative quests. A user selects a goal, vibe, duration, budget, difficulty, date/time, and solo/multiplayer mode; the system generates a quest arc, guides the player through checkpoints, validates progress using device signals, and issues a reward or payment action at completion.


Core concept:
- "Dungeons & Dragons x Pokemon Go x Strava x Skyrim" for real life.
- The quest is not only text generation; it is an orchestrated sequence of real-world actions, checkpoints, media outputs, and optional external actions.
- The narrative may adapt during execution, but the quest runtime must remain bounded by a deterministic state machine.


Primary user goals:
- sport
- culture
- discovery
- music
- learning a specific skill
- travel
- meeting new people
- fun


## 2. Hackathon MVP scope


We ship exactly one end-to-end single-player mobile quest for demo day.


MVP includes:
- mobile onboarding
- Dynamic wallet/auth onboarding
- one generated quest
- 3 checkpoints maximum
- one live agent loop during quest execution
- one proof mechanism per checkpoint
- one Hedera reward/payment event at quest completion
- one post-quest recap screen


MVP excludes:
- multiplayer sync
- generalized booking engine
- social graph
- tradable inventory marketplace
- complex NFT economy
- dependency on glasses for core completion
- arbitrary open-ended agent autonomy
- many quest templates


Demo constraint:
The demo must run in 5 minutes, even if the product vision supports longer quests such as 4 hours.


Success criteria:
- user launches quest in under 60 seconds
- user completes 3 checkpoints in demo mode
- system verifies at least 2 checkpoints automatically
- one Hedera-linked reward/payment action is triggered
- recap screen renders final grade + reward result


## 3. User flow


### 3.1 Onboarding
User opens mobile app, connects device context, initializes wallet/auth via Dynamic, and optionally connects glasses if available.


### 3.2 Quest launch
User provides:
- goal
- vibe
- duration
- budget
- location
- difficulty
- players
- datetime


System then:
1. builds city context
2. generates quest plan
3. persists quest state
4. starts the first checkpoint


### 3.3 During quest
For each checkpoint:
1. client displays objective + narrative context
2. user moves to location or performs action
3. client submits event/proof
4. backend verifies proof
5. quest engine advances or requests replan
6. agent returns updated narrative text / next step


### 3.4 After quest
System displays:
- completion status
- grade / after-action review
- best clip or recap artifact
- reward/payment status
- optional preview of next quest


## 4. Technical architecture


Architecture principle:
LLM generates plans and narrative, but quest execution is controlled by backend state and event processing.


### 4.1 Runtime services


1. Mobile client
   - stack: React Native / Expo preferred
   - responsibilities: onboarding, quest UI, proof capture, wallet session, recap
   - local responsibility: cache active quest and queue events during transient network failure


2. API service
   - stack: FastAPI
   - responsibilities: public API, auth context propagation, quest/session bootstrap, media upload URLs, orchestration entrypoints


3. Quest engine
   - deterministic state machine
   - owns checkpoint lifecycle, timers, branch eligibility, failure handling, reward eligibility


4. Agent orchestration service
   - LLM-driven planning + narrative generation
   - contains city research pass, quest draft pass, targeted validation pass, final quest compiler
   - may request tools, but cannot directly mutate quest state without backend confirmation


5. Event ingestion / verification service
   - normalizes GPS, camera, microphone, transcript, and manual override events
   - verifies checkpoint conditions
   - emits canonical quest events


6. Sponsor adapters
   - Dynamic adapter: mobile wallet/auth bootstrap
   - Hedera adapter: agentic payment / reward issuance
   - 0G/OpenClaw-oriented adapter: agent execution layer / sponsor-track integration surface


### 4.2 High-level system diagram


Client App
  -> API Gateway
    -> Quest Engine
    -> Agent Service
       -> City Research Agent
       -> Quest Writer / Replanner
    -> Event Verifier
    -> Reward Service
       -> Hedera
    -> Auth / Wallet Adapter
       -> Dynamic


### 4.3 Source of truth


Backend is the source of truth for:
- active quest state
- checkpoint status
- event history
- proof verification result
- reward status


Client cache is advisory only.


## 5. Sponsor mapping


### Dynamic
Used for mobile-first auth and wallet bootstrap. Dynamic must appear in the critical path of onboarding, not as an optional add-on.


### Hedera
Used for one visible onchain action tied to quest completion. For MVP, this can be:
- completion reward issuance
- payment trigger
- achievement settlement
- badge / receipt-like completion record


### 0G
Used as the sponsor-facing agent track. The submission should show a real agentic execution layer, not only text generation inside the app.

## 7. Quest state machine


Canonical states:
- DRAFT
- RESEARCHING
- GENERATED
- READY
- ACTIVE
- CHECKPOINT_PENDING
- PROOF_SUBMITTED
- CHECKPOINT_VERIFIED
- REPLAN_REQUIRED
- REWARD_PENDING
- COMPLETED
- FAILED
- ABORTED


State transition rules:
- only Quest Engine may change quest state
- Agent Service may propose updates, never commit them directly
- Reward state is independent but linked to terminal quest states
- proof verification is asynchronous but must resolve before checkpoint completion

## 9. Agentic orchestration


### 9.1 Design principle
Quest generation is split into bounded passes instead of one monolithic prompt.


### 9.2 Passes


Pass 1 — Broad research
- generate 5-8 search queries from user inputs
- pull broad candidate options from web/search sources
- build a rough pool of activities, events, restaurants, POIs, transport context


Pass 2 — Draft quest
- transform city options into a provisional arc
- define tentative checkpoints
- identify missing data such as price, hours, distance, feasibility


Pass 3 — Targeted research
- query exact hours, prices, route times, fallback venues
- validate feasibility under budget, date/time, and travel constraints


Pass 4 — Final quest/context compile
- output structured CityContext
- output structured QuestPlan JSON
- generate briefing package


This pass-based design is already the most concrete part of the current agent architecture and should remain the foundation of MVP quest generation.


### 9.3 Research agent implementation


Current implementation assumptions:
- Python + FastAPI backend
- Anthropic API as orchestrator
- tool-use loop with bounded iterations
- tool layer for generic search, TripAdvisor, Google Maps-style place lookup, Luma, GetYourGuide, weather, directions
- Pydantic models for QuestRequest and CityContext
- `/research` endpoint returning structured CityContext


### 9.4 Agent boundaries


The City Research Agent is responsible for:
- gathering real-world options
- structuring them into CityContext
- returning ranked and feasible candidates


It is not responsible for:
- directly issuing rewards
- mutating live quest state
- final checkpoint verification
- bypassing budget/time constraints


## 10. Research data contract


```python
class CityContext(BaseModel):
    location: LocationInfo
    activities: list[Activity]
    restaurants: list[Restaurant]
    events: list[Event]
    points_of_interest: list[POI]
    transport: TransportInfo
    raw_notes: str
```


Minimum output requirements:
- at least 5 candidate activities
- at least 3 location anchors / POIs
- at least 2 backup options
- transport notes
- raw notes including unresolved uncertainty


Quest generation must fail closed if CityContext quality is too low.

## 12. Proof and verification


Supported MVP proof types:
- GPS geofence
- image capture
- voice confirmation
- manual demo override


Rule:
Each checkpoint has exactly one primary proof type and one fallback only.


Verification policy:
- GPS: radius-based match
- image: server-side or heuristic confirmation; can be human-assisted in demo
- voice: keyword / phrase confirmation
- manual override: internal admin/demo tool only


## 13. APIs


Minimal endpoints:
- `POST /research`
- `POST /quests/generate`
- `POST /quests/{id}/start`
- `GET /quests/{id}`
- `POST /quests/{id}/events`
- `POST /quests/{id}/proof`
- `POST /quests/{id}/replan`
- `POST /quests/{id}/reward`
- `GET /health`


Example responsibility split:
- `/research` -> City Research Agent
- `/quests/generate` -> Quest writer + Quest Engine bootstrap
- `/events` -> event ingestion
- `/proof` -> verification pipeline
- `/reward` -> Hedera adapter


## 14. Failure handling


Required fallbacks:
- no glasses -> phone-only flow
- bad GPS -> manual code or image fallback
- search agent incomplete -> use backup curated Cannes template
- agent timeout -> continue from precomputed quest branch
- payment/reward failure -> mark reward pending and retry async
- low-confidence place validation -> manual demo override


System must never block the entire quest on one flaky external tool.


## 15. Security / privacy


Sensitive inputs:
- GPS
- camera/media
- microphone/transcript
- wallet identity
- optional payment action


MVP rules:
- do not store unnecessary raw media permanently
- store only metadata and demo artifacts needed for recap
- log all reward actions
- keep user-facing permissions explicit at onboarding

## 17. Demo scenario


Recommended demo:
- location: Cannes old town / nearby fixed landmarks
- duration: 5-minute compressed mode
- 3 checkpoints
- checkpoint 1: GPS arrival
- checkpoint 2: image or spoken phrase
- checkpoint 3: final interaction / confirmation
- completion: Hedera reward action
- recap: grade + artifact + next-quest tease


## 18. Build rule


The architecture must stay event-driven and state-machine-first. The agent layer can enrich planning, narrative, and replanning, but deterministic services own execution, verification, and settlement.
